# Deploy-First Loop (MVP)

유저 리서치를 먼저 크게 하지 않고, `deploy -> 계측 -> 가설 검증 -> 수정` 루프로 MVP를 발전시키기 위한 최소 운영 규칙.

## 기본 원칙

- 실험 1개는 변경 1개를 기본값으로 한다.
- 기능은 "가설을 검증하기 위한 도구"로만 만든다.
- 배포 전 성공/실패 기준을 문서로 먼저 고정한다(사후 합리화 방지).
- 코어 지표/이벤트 정의는 먼저 확정하고, 구현은 뒤따라간다.

## 루프(권장 주기: 가능하면 매일)

1. 이번 배포의 "한 가지 변화"를 고른다(필요 시: `context/core/hypotheses.md`).
2. 성공/실패 기준을 최소 1개 숫자로 적는다(필요 시: `context/core/metrics.md`).
3. 필요한 계측 이벤트를 정의한다(필요 시: `context/analytics/events.md`).
4. 변경 1개를 구현하고 배포한다.
5. 결과를 짧은 로그로 남긴다(선택: `context/experiments/`).
6. 큰 방향 전환/스코프 컷 같은 결정만 결정 로그로 남긴다(선택: `context/decisions/`).

## Non-goals (MVP 단계)

- 완벽한 문서화
- 완벽한 실험 설계(표본/통계)
- 모든 가설을 한 번에 검증

## 체크리스트 (배포 전)

- 이번 배포에서 바뀌는 "한 가지"가 명확한가?
- 성공/실패 기준이 문서로 고정돼 있는가?
- 코어 이벤트가 실제로 찍히는지 로컬/스테이징에서 확인했는가?
- 실패 시 다음 배포(대안)이 이미 정리돼 있는가?
